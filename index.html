GhostTap

We introduce a novel class of tapjacking attacks that undermine user trust by creating phantom windows—malicious overlays that intercept user input while evading Android's security mechanisms. This site presents our research, key findings, and live demonstrations.

GhostTap

Four complementary attack surfaces

Android's complex GUI architecture causes state inconsistencies across components during asynchronous IPC. We identify and exploit four distinct attack scenarios.

01

GhostIPC

Exploits asymmetric IPC buffer limits (Sync vs. Oneway) to trigger selective failures, desynchronizing state between system_server and SystemUI.

  • Bypasses Background Activity Launch (BAL)
  • Affects Android 14
02

GhostWindow

Exploits ONEWAY buffer overflow during VSYNC updates to desynchronize SurfaceFlingers (rendering) from InputDispatcher (routing).

  • Bypasses Obscured Touch Detection, Trusted Overlay, ARIS, Anti-Overlay Flags
  • Affects Android 12–16
03

GhostARIS

Exploits the animation state exemption in ARIS, causing InputDispatcher to skip input filtering while an Activity is animating.

  • Bypasses ARIS and Anti-Overlay Flags
  • Affects Android 12–16
04

GhostTrust

Exploits privilege state inconsistencies during PiP transition failures to retain TRUSTED_OVERLAY status on attacker-controlled windows.

  • Bypasses Obscured Touch Detection, Trusted Overlay, ARIS, Anti-Overlay Flags
  • Affects Android 12–16

Why it matters

UI Integrity Requires Holistic Defenses

GhostTap demonstrates that UI, IPC, and trust cues must be protected together. Point solutions (e.g., overlay warnings) are insufficient when windows, gestures, and intents can be combined.

Cross-Layer Exploits

UI overlays, window races, and IPC relays can be chained for real-world hijacks.

User Deception

Attacks succeed without root, special permissions, or visible artifacts.

Actionable Fixes

We recommend stricter window focus rules, improved intent verification, and stronger UI integrity signals.
